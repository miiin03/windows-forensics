# WinTrace vs 무료 도구 대조실험 가이드 (Windows Timeline 카빙)

> **대상 독자**: 같은 팀원 (실험 직접 수행)
> **소요 시간**: 환경 구축 1\~2시간 + 실험 1시간
> **OS**: Windows 10/11 (테스트는 Windows에서, 우리 도구는 Python만 있으면 OS 무관)
> **목표**: 공개 데이터셋에서 무료/오픈소스 포렌식 도구와 **WinTrace**(`windows-timeline` 엔진)를 같은 조건으로 돌려 **우리 도구의 우위**(삭제 레코드 복원 수, 스키마 인지 디코딩, 통합성, 무결성)를 정량/정성으로 입증한다.

---

## 1. 목적 — 왜 이 실험인가

`ActivitiesCache.db`(Windows Timeline)는 SQLite DB다. 무료 도구들은 각각 한 가지만 한다:

- **WxTCmd** (Eric Zimmerman): 살아있는(live) 레코드만 CSV로 파싱. **삭제 복원 없음.**
- **FQLite / Undark**: SQLite 내부 미할당 영역에서 삭제 레코드를 **카빙**하지만, **ActivitiesCache 스키마를 모른다.** raw 셀/행을 그대로 토해낸다.

**WinTrace**는 이 둘을 합치고 더 간다:

1. **live 파싱** + **삭제 레코드 카빙**을 한 번에 — SQLite 내부 미할당 4개 소스(① freelist 페이지, ② in-page free block, ③ cell slack, ④ WAL 파일)에서 복원.
2. 31컬럼 `Activity` 테이블을 **의미 단위로 디코드**: Id(GUID), AppId→앱 이름, Payload(JSON 파싱), 타임스탬프→ISO/KST, 소스+신뢰도(confidence), provenance(페이지/오프셋/byte range).
3. **read-only(immutable)** 로 증거를 건드리지 않고 실행, 통합 JSON 출력.

이 실험으로 보여줄 것:

| 축 | 주장 | 측정 방법 |
|---|---|---|
| 라이브 정확도 | WxTCmd와 **동일한** live 레코드 수·필드 | 레코드 수·Id·AppId·타임스탬프 1:1 대조 |
| 복원력 | FQLite/Undark **이상으로** 삭제 레코드 복원 | ground-truth 대비 복원 수, 커버 소스 수 |
| 스키마 인지 | 무료 카버는 raw bytes, 우리는 구조화 레코드 | 같은 레코드의 raw vs 디코드 side-by-side |
| 통합성 | 무료는 단일 목적, 우리는 Timeline+EVTX+AI 통합 | 정성 비교 |
| 무결성 | read-only + 해시, 재현성 | 절차 기록 |

## 2. 데이터셋 준비 (공개 GitHub/NIST 데이터 + 알려진 삭제 만들기)

> **원칙(팀 확정)**: 데이터셋 2종 고정 — EVTX는 공개 GitHub(EVTX-ATTACK-SAMPLES)를 **그대로**, ActivitiesCache는 1102 증거인멸과 연결되도록 **팀이 만든 합성 데모 DB**를 쓴다. 공개 ActivitiesCache 바이너리는 존재하지 않으므로(확인됨), 합성 DB가 오히려 *알려진 정답(ground-truth)* 대비 카빙 복원율 측정에 적합하다.

### 2-A. 데이터셋 (팀 확정)

| 아티팩트 | 소스 | 비고 |
|---|---|---|
| **EVTX 이벤트로그** | **EVTX-ATTACK-SAMPLES** — https://github.com/sbousseaden/EVTX-ATTACK-SAMPLES (공개, 그대로 사용) | 특히 `DE_1102_security_log_cleared.evtx` (EventID **1102 = 보안 감사로그 삭제 = 증거 인멸**) |
| **ActivitiesCache.db** | **팀 합성 데모 DB** — `wintrace/sample/build_demo_db.py` (증거인멸용 웹검색 시도 등 활동을 담아 1102와 연결) | 공개 ActivitiesCache 바이너리 없음 → 재현 가능한 합성 DB가 정답 측정에 적합 |

> **스토리**: 공격자가 "로그 삭제·증거 인멸" 관련 **웹 검색**을 한 흔적이 ActivitiesCache에 남고(삭제해도 carving으로 복원) + EVTX **1102**가 실제 보안로그 삭제를 증명 → 통합 타임라인이 **은폐 시도 전체**를 같은 시각대로 재구성.

### 2-B. 베이스 DB 생성 + EVTX 준비

**ActivitiesCache (팀 합성 데모 DB — 발표 데모와 동일):**
```bash
cd wintrace/sample
python build_demo_db.py        # → ActivitiesCache.db (증거인멸 웹검색 스토리 + 데모 활동)
```
> 대조 실험과 발표가 **한 데이터셋**으로 일관됨.

**EVTX (공개, 그대로):**
```bash
git clone https://github.com/sbousseaden/EVTX-ATTACK-SAMPLES.git
# 핵심 입력 파일:
#   ".../Defense Evasion/DE_1102_security_log_cleared.evtx"   (1102 증거인멸)
# 우리 도구 ↔ EvtxECmd(Eric Zimmerman, 무료) 둘 다 이 .evtx 를 입력으로 대조
```
> EVTX 대조 도구 = **EvtxECmd**(무료): `EvtxECmd.exe -f DE_1102_security_log_cleared.evtx --csv out`. 우리 파서의 EventID/시각/계정 추출이 EvtxECmd와 일치하는지 검증.

### 2-C. 재현 가능한 삭제 만들기 + ground-truth 기록 (★핵심)

카빙 복원율을 *정답 대비* 측정하려면 "무엇을 지웠는지"를 알아야 한다. **`secure_delete`를 OFF로 두고**(안 그러면 삭제 바이트가 0으로 덮여 카빙 불가) N개를 지운 뒤, 지운 Id 목록을 정답으로 보관한다.

```powershell
cd C:\work\bench\dataset
# 0) 작업 사본 (원본 절대 건드리지 않음)
Copy-Item ActivitiesCache.db ActivitiesCache.deleted.db -Force

# 1) 삭제 전 ground-truth 기록 (전체 Id 스냅샷)
sqlite3 ActivitiesCache.deleted.db ".mode csv" ".headers on" `
  "SELECT rowid, hex(Id) AS Id, AppId FROM Activity ORDER BY rowid;" > ground_truth_all.csv

# 2) secure_delete OFF + 알려진 N개 삭제 (예: rowid 5,7,9,11,13 = 5개)
sqlite3 ActivitiesCache.deleted.db @"
PRAGMA secure_delete=OFF;
PRAGMA journal_mode=WAL;            -- WAL 소스도 실험하려면 WAL 모드 유지
.mode csv
.headers on
.once deleted_ground_truth.csv
SELECT rowid, hex(Id) AS Id, AppId FROM Activity WHERE rowid IN (5,7,9,11,13);
DELETE FROM Activity WHERE rowid IN (5,7,9,11,13);
"@
# 3) 결과 확인: 어떤 삭제가 어떤 소스에 남는지
#  - 일부는 freelist 페이지로, 일부는 in-page free block / cell slack 으로,
#  - WAL 모드면 직전 버전이 -wal 파일에 남는다 (checkpoint 전).
```

> **정답 파일 3종을 커밋/보관**:
> - `ground_truth_all.csv` — 삭제 전 전체 (live 정확도 검증용)
> - `deleted_ground_truth.csv` — 지운 5개의 Id (복원율 분모)
> - 삭제 명령 자체(위 스크립트) — 재현용
>
> **주의**: `VACUUM`이나 `PRAGMA secure_delete=ON`을 절대 돌리지 마라. 둘 다 미할당 영역을 밀어 카빙을 무력화한다(§7 참조). WAL 소스를 보려면 삭제 후 **checkpoint하지 말 것**.

이제 실험 입력은 **`ActivitiesCache.deleted.db`**(+ 있으면 `-wal`)이고, 정답은 위 CSV 3종이다.

## 3. 도구 설치 (복붙 가능)

작업 루트: `C:\work\bench`. 모든 출력은 `C:\work\bench\out\<도구>` 에 저장.

```powershell
New-Item -ItemType Directory -Force -Path C:\work\bench\out\wxtcmd,C:\work\bench\out\fqlite,C:\work\bench\out\undark,C:\work\bench\out\wintrace | Out-Null
```

### 3-1. WxTCmd (Eric Zimmerman, .NET) — live 파서

- 다운로드: https://ericzimmerman.github.io/ (Get-ZimmermanTools 또는 직접 zip) / 저장소 https://github.com/EricZimmerman/WxTCmd
- 요구: .NET (최신 빌드는 .NET 6+; ezTools 페이지에서 net6 버전 받기)

```powershell
# 권장: 공식 일괄 다운로더
iwr https://raw.githubusercontent.com/EricZimmerman/Get-ZimmermanTools/master/Get-ZimmermanTools.ps1 -OutFile C:\work\bench\Get-ZimmermanTools.ps1
powershell -ExecutionPolicy Bypass -File C:\work\bench\Get-ZimmermanTools.ps1 -Dest C:\work\bench\tools
# → C:\work\bench\tools\...\WxTCmd.exe 생성
```

### 3-2. FQLite (Java) — SQLite 카버 (freelist + WAL + rollback journal)

- 저장소: https://github.com/pawlaszczyk/fqlite
- 릴리스: https://github.com/pawlaszczyk/fqlite/releases (최신 **4.x**, 2026년 6월 기준 v4.22)
- **중요**: v2.0부터 **CLI 모드 폐지 → GUI 전용**(JavaFX). 최신 빌드는 **JRE 번들** 포함이라 별도 Java 설치 불필요.
- Windows: 릴리스의 **`.exe`/`.msi` 설치 프로그램** 실행 → 시작 메뉴에서 실행.

```powershell
# 릴리스 페이지에서 Windows 설치 파일 받아 실행 (브라우저)
#   https://github.com/pawlaszczyk/fqlite/releases  →  fqlite_<버전>_windows .exe/.msi
# 설치 후 GUI 실행 → File > Open DB → ActivitiesCache.deleted.db 선택
# (동반 -wal/-journal 파일이 같은 폴더에 있으면 자동 로드됨)
```
> 구버전 CLI가 필요하면 v1.x 릴리스를 받을 수 있으나, **최신 GUI 버전으로 측정**하고 결과 화면을 export/스크린샷으로 캡처하는 것을 권장.

### 3-3. Undark (C++ CLI) — SQLite freelist/미할당 카버

- 저장소(주): https://github.com/alitrack/undark — `undark.c`, `varint.c/.h`, `Makefile`, `Makefile.mingw32`
- 출력: **plain text CSV** (visible + deleted 행 모두 덤프)

```bash
# Linux/WSL/MSYS2
git clone https://github.com/alitrack/undark.git
cd undark
make                      # gcc 빌드 → ./undark
# Windows 네이티브 빌드 (MinGW):
#   mingw32-make -f Makefile.mingw32   → undark.exe
```

```powershell
# 실행 (deleted 행까지 CSV로)
.\undark.exe -i C:\work\bench\dataset\ActivitiesCache.deleted.db > C:\work\bench\out\undark\undark_all.csv
# 세밀 탐색(바이트 단위) 옵션:
.\undark.exe -i ...\ActivitiesCache.deleted.db --fine-search > ...\undark_fine.csv
```
사용법 전체: `undark -i <db> [-d] [-v] [--cellcount-min/max] [--rowsize-min/max] [--no-blobs] [--blob-size-limit] [--fine-search]`

### 3-4. (선택) DB Browser for SQLite — live 베이스라인

- https://sqlitebrowser.org/ — DB 열어 `SELECT count(*) FROM Activity;`로 live 레코드 수 눈으로 확인(삭제 복원 기능 없음, 기준선 용도).

### 3-5. 우리 도구 — WinTrace (`windows-timeline`)

```powershell
# 저장소 루트에서
cd <repo>\windows-timeline
# (가상환경 권장) python -m venv .venv ; .\.venv\Scripts\activate
```

> 4개 카빙 소스 전부 보려면: `python -m engine ...deleted.db --phase 3 -o ...\result_full.json --pretty`

## 4. 실험 절차 (실행 명령 + 출력 위치)

### 4-0. 무결성 기준선 (먼저)

```powershell
# 입력 DB 해시 기록 (실험 전후 동일해야 = read-only 입증)
Get-FileHash C:\work\bench\dataset\ActivitiesCache.deleted.db -Algorithm SHA256 | Tee-Object C:\work\bench\out\hash_before.txt
```

### 4-1. WxTCmd — live 파싱 → CSV

```powershell
& "C:\work\bench\tools\...\WxTCmd.exe" -f "C:\work\bench\dataset\ActivitiesCache.deleted.db" --csv "C:\work\bench\out\wxtcmd"
# 출력: out\wxtcmd\*_Activity.csv  → live Activity 레코드. 행 수 = live 레코드 수.
```
캡처: 생성된 CSV 행 수, `Id`/`Executable(AppId)`/`StartTime`/`LastModified` 열.

### 4-2. Undark — 카빙 → CSV

```powershell
.\undark.exe -i "C:\work\bench\dataset\ActivitiesCache.deleted.db" > C:\work\bench\out\undark\undark_all.csv
.\undark.exe -i "C:\work\bench\dataset\ActivitiesCache.deleted.db" --fine-search > C:\work\bench\out\undark\undark_fine.csv
```
캡처: 전체 행 수, 그중 deleted로 추정되는 행, `deleted_ground_truth.csv`의 5개 Id가 raw 바이트로 몇 개 나오나.

### 4-3. FQLite — 카빙 (GUI)

1. FQLite 실행 → File > Open Database → `ActivitiesCache.deleted.db`
2. `Activity` 테이블 노드 선택 → 상태가 **deleted**(보통 빨강)인 행 수 세기
3. 어떤 소스에서 왔는지 확인: **freelist / WAL / rollback journal** 탭/표시
4. 결과를 **Export(CSV)** 또는 스크린샷 → `out\fqlite\`

캡처: deleted 복원 행 수, 커버 소스, 5개 정답 중 몇 개 복원.

### 4-4. WinTrace — live+카빙 통합

```powershell
cd <repo>\windows-timeline
# 전체 소스 + pretty
python -m engine "C:\work\bench\dataset\ActivitiesCache.deleted.db" --phase 3 -o "C:\work\bench\out\wintrace\result_full.json" --pretty
# 통계만 빠르게
python -m engine "C:\work\bench\dataset\ActivitiesCache.deleted.db" --phase 3 --stats-only -o "C:\work\bench\out\wintrace\stats.json" --pretty
```
JSON에서 읽을 값(export.py `stats` 기준):
- `stats.live` — live 레코드 수 (→ WxTCmd와 대조)
- `stats.carved_total`, `stats.by_source`(freelist/freeblock/wal/slack 분포) — 복원 수·소스
- `stats.deleted_recovered`, `stats.prior_versions` — 삭제 복원/WAL 직전버전
- 각 record: `is_deleted`, `confidence`/`confidence_label`, `provenance{page_no,cell_offset,byte_range}`, 디코드된 `Id`,`app_name`,`timestamps{epoch,iso8601}`,`Payload{parsed}`

```powershell
# 5개 정답 중 복원 수 빠른 집계 (PowerShell)
$gt = Import-Csv C:\work\bench\dataset\deleted_ground_truth.csv | % { $_.Id.ToLower() }
$r  = Get-Content C:\work\bench\out\wintrace\result_full.json -Raw | ConvertFrom-Json
$ids = $r.records | ? { $_.is_deleted } | % { ($_.Id -replace '[^0-9a-f]','').ToLower() }
($gt | ? { $ids -contains $_ }).Count    # = 복원된 정답 수
```

```powershell
Compare-Object (gc C:\work\bench\out\hash_before.txt) (gc C:\work\bench\out\hash_after.txt)
```

## 5. 결과표 템플릿 (빈칸 채우기)

> 같은 입력(`ActivitiesCache.deleted.db`, 삭제 5개 = `deleted_ground_truth.csv`)에서 측정.

### 5-1. 핵심 비교표

| 도구 | 라이브 레코드수 | 복원 삭제수 (/5) | 커버 소스 (freelist/freeblock/slack/WAL) | 스키마 디코드 | 비고 |
|---|---|---|---|---|---|
| WxTCmd | ___ | 0 (복원 미지원) | — | live만 (CSV) | live 정확도 기준선 |
| Undark | ___ (visible 덤프) | ___ | freelist/미할당 (slack·freeblock ✗, WAL ✗) | ✗ raw CSV | byte→스키마 의미 없음 |
| FQLite | ___ | ___ | freelist / WAL / journal (slack·freeblock ✗) | ✗ raw row | GUI, export |
| **WinTrace (우리)** | ___ | ___ | **✓ ✓ ✓ ✓ (4개 전부)** | **✓ 의미 디코드** | read-only, 통합 JSON |

> 기대: `WinTrace 복원수 ≥ FQLite, Undark`. live 수는 `WxTCmd == WinTrace.stats.live` 여야 정상.

### 5-2. live 정확도 1:1 대조 (WxTCmd vs WinTrace)

| 항목 | WxTCmd CSV | WinTrace `stats.live`/records | 일치? |
|---|---|---|---|
| 레코드 수 | ___ | ___ | ☐ |
| Id (GUID) 집합 | ___ | ___ | ☐ |
| AppId/Executable | ___ | ___ | ☐ |
| StartTime/LastModified | ___ | ___ | ☐ |

### 5-3. 같은 복원 레코드 — raw vs 디코드 (스키마 인지 우위)

| 필드 | FQLite/Undark (raw) | WinTrace (디코드) |
|---|---|---|
| 식별자 | `x'00112233...EEFF'` (raw BLOB) | `Id: "00112233445566778899aabbccddeeff"` |
| 앱 | `[{"application":"WINWORD.EXE"}]` 원문 | `app_name: "Microsoft Word"`, `AppId_raw` 보존 |
| Payload | raw BLOB/문자열 | `Payload.parsed{...}` (JSON 파싱, displayText 등) |
| 타임스탬프 | epoch 정수 그대로 | `timestamps.StartTime{epoch, iso8601}` + KST |
| 소스/신뢰도 | 없음 | `source: freelist`, `confidence_label: medium` |
| 위치 증명 | 없음(또는 페이지만) | `provenance{page_no, cell_offset, byte_range}` |

(실제 값은 본인 데이터로 채워 넣어 한 행을 완성)

---

## 6. 분석 포인트 (우리 우위 강조)

1. **복원 수 (정량)**: `WinTrace.stats.deleted_recovered ≥ FQLite ≥ Undark`. 특히 **slack·in-page free block·WAL 직전버전**은 무료 카버가 못 잡거나(slack/freeblock) 부분만 잡는 영역 — 우리는 4개 소스 전부 커버. `stats.by_source`로 소스별 기여를 그대로 제시.
2. **스키마 인지 (정성+정량)**: 무료 카버는 raw 셀/CSV. 우리는 `Id`(GUID 정규화)·`app_name`(AppId→앱명)·`Payload.parsed`(JSON)·`timestamps.iso8601`+KST·`confidence`까지 디코드 → 분석가가 즉시 읽는다. 5-3 표 한 행이 결정적 증거.
3. **provenance/재현성**: 각 복원 레코드에 `page_no/cell_offset/byte_range/serial_types/decode_errors`가 붙어 **법정 추적성** 확보. 무료 도구엔 없음.
4. **통합성**: 무료 도구는 단일 목적(파싱 1개 or 카빙 1개). WinTrace는 Timeline 카빙 + EVTX + AI + 통합 타임라인을 **오프라인**으로 묶는다 — 도구 3\~4개를 손으로 엮을 필요 없음.
5. **무결성**: `hash_before == hash_after` + immutable 오픈 → read-only 입증. 절차/해시/ground-truth를 모두 기록해 재현 가능.

## 7. 한계·주의

- **`secure_delete=ON` 또는 `VACUUM`**: 삭제 바이트가 0으로 덮이거나 미할당이 회수됨 → **모든 카버(우리 포함) 복원 불가**. 실험 DB엔 절대 적용 금지. (Windows의 실제 ActivitiesCache는 보통 secure_delete가 아니라 카빙 가능.)
- **WAL checkpoint**: 삭제 후 checkpoint/`wal_checkpoint(TRUNCATE)` 하면 WAL 직전버전 소스가 사라짐. WAL 소스를 측정하려면 checkpoint 전 상태로 둘 것.
- **공정성**: FQLite는 최신 GUI 버전, Undark는 `--fine-search` 포함 두 번 돌려 **각 도구의 최대 복원치**로 비교(우리에게 유리하게 깎지 않기). WxTCmd는 복원 도구가 아니므로 "복원 0"은 한계 명시이지 비방이 아님 — live 정확도 기준선으로만 사용.
- **데이터셋 한계**: 공개 timeline DB 바이너리가 없어 베이스를 합성/추출했다 — 결과는 "구조·복원 메커니즘" 비교로 일반화 가능하나, 절대 수치는 사용한 DB에 의존. **CFReDS 이미지 추출본으로 1회 더 재현**하면 신뢰도가 올라간다.
- **버전 드리프트**: WxTCmd/.NET, FQLite v4.x, Undark 빌드는 시점에 따라 바뀐다 — 실험 시 각 도구 버전을 결과표 비고에 적어라.
